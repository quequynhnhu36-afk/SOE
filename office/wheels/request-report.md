# Request and Report Wheel

## 适用场景

用于请示、报告、呈批件、事项说明、申请材料、向上级单位报送情况说明。

请示用于请求批准或协调。报告用于说明情况、反馈结果、报备进展。不要混用。

## 必要输入

| 输入 | 说明 |
| --- | --- |
| 文种 | 请示、报告、呈批件、事项说明、申请 |
| 报送对象 | 集团、上级部门、领导、外部单位 |
| 事项背景 | 为什么提出该事项 |
| 具体请求或报告内容 | 请求批准什么，或报告什么进展 |
| 依据 | 制度、会议要求、工作计划、合同、政策，缺失则标 `[待补充依据]` |
| 风险和影响 | 不办理的影响、需注意的问题 |
| 需审批事项 | 金额、资源、授权、人员、时间 |

## 工作步骤

1. 判断文种：有明确审批诉求用请示；只说明情况用报告。
2. 用第一段写清事项背景和目的。
3. 用主体部分说明事实、依据、方案和影响。
4. 请示必须明确“拟请批准事项”，不要让领导猜。
5. 报告必须说明已完成、正在推进、存在问题和下一步。
6. 对缺少依据、金额、附件的内容使用 `[待补充]`。
7. 结尾使用规范语句。

## 输出结构

### 请示

1. 标题：关于 `[事项]` 的请示
2. 主送单位
3. 背景和必要性
4. 主要内容和依据
5. 风险影响和保障措施
6. 拟请批准事项
7. 结束语：妥否，请批示。

### 报告

1. 标题：关于 `[事项]` 有关情况的报告
2. 主送单位
3. 基本情况
4. 工作进展
5. 存在问题
6. 下一步安排
7. 结束语：特此报告。

## 禁止事项

- 不把“报告”写成变相请示。
- 不模糊审批事项。
- 不编造制度依据、会议纪要或预算来源。
- 不替用户承诺无法确认的完成期限。
- 不写真实敏感审批链路到公开样例中。

## 质量自检

- 文种是否正确。
- 审批或报送对象是否清楚。
- 背景、依据、事项、影响是否完整。
- 请示是否明确“请批事项”。
- 报告是否说明进展、问题和下一步。
- 缺失信息是否用 `[待补充]` 标明。

## 示例任务

输入：

> 写一个请示，请集团同意我们开展数字化台账试点，需要 3 个月，先覆盖 6 家单位，没有预算信息。

输出应包含：

- 标题：关于开展数字化台账试点工作的请示
- 试点背景和必要性
- 试点范围：6 家单位
- 周期：3 个月
- 预算：`[待补充预算安排]`
- 拟请批准事项：同意开展试点，并协调相关单位配合
